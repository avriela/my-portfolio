Telegram-бот учёта товаров и продаж
Бот для учёта закупок, складских остатков и продаж электроники и комплектующих. Импорт данных из Excel, автоматические расчёты себестоимости, выгрузка отчётов и уведомления при низком остатке.
О продукте

Бот работает в личных сообщениях и поддерживает импорт данных из Excel с автоматическим расчётом себестоимости с учётом курса закупочной валюты, учёт складских остатков, фиксацию продаж двух типов и выгрузку отчётов по продажам, доставке и остаткам.

При падении остатка товара ниже порогового значения бот отправляет уведомление. Управление через инлайн-клавиатуру: четыре основных раздела — номенклатура заказов, номенклатура прихода, продажи, статистика.

Моя роль и задача

Тестирование шло итерациями. Находила баги, часть передавала разработчику сразу в переписке, часть накапливала в структурированный репорт и передавала пакетом. После каждого фикса проводила ретест.

Часть поведения в процессе оказывалась не багами: некоторые сценарии были задуманы именно так, часть ограничений объяснялась особенностями самого Telegram. Разработчик пояснял — я фиксировала и не возвращалась к этому повторно.

Что тестировала и как
Основные сценарии

Проверяла полный пользовательский путь по каждому разделу: загрузка файла, навигация между шагами, корректность кнопок, поведение при отмене сценария.

Часть проблем всплыла уже на этом уровне: кнопки не запускались, бот не отвечал, навигация работала не так как описано в ТЗ.

Импорт Excel — валидация данных

Основной объём тестирования. Проверяла поведение бота при различных входных данных:

  • корректный файл в ожидаемом формате
  • файл с переименованными или англоязычными столбцами
  • отсутствующие обязательные столбцы
  • пустые ячейки в обязательных полях
  • дробные значения там где ожидаются целые
  • нулевые и отрицательные значения в числовых полях
  • дублирующиеся позиции с разными атрибутами
  • большой объём данных с повторяющимися позициями

Для каждого случая смотрела что бот возвращает пользователю: есть ли сообщение об ошибке, насколько оно информативно, соответствует ли поведение ТЗ.

Проверка логики подсчётов

Брала выгрузку из бота и проверяла цифры вручную — по калькулятору и формулам из ТЗ: себестоимость единицы, итоговые суммы, прибыль по периодам. Сверяла не только итоговые цифры но и промежуточные.

Отдельно проверяла поведение расчётов при некорректных входных данных: нулевых количествах, отрицательных ценах, пустых полях. Смотрела отражаются ли граничные ситуации корректно в сообщениях бота.

Статистика и выгрузки

Проверяла три типа выгрузок: продажи, ожидаемая доставка, складские остатки. Сверяла с ТЗ состав колонок, корректность данных, отображение временных меток.

Воспроизводила ситуации с дублирующимися позициями и смотрела как бот их обрабатывает в отчётах.

Сценарии продаж

Прорабатывала сценарии по каждому источнику товара: позиции со склада и прямые закупки без складского учёта. Проверяла валидацию на каждом шаге, поведение опциональных полей, корректность списания остатков, поведение кнопки отмены в середине сценария.

Примеры найденных багов
  • Числовые поля принимали отрицательные и нулевые значения без ошибки
  • Дробные значения принимались там где по ТЗ допустимы только целые
  • При продаже товара в количестве большем чем есть на складе бот не блокировал операцию и уходил в отрицательный остаток
  • При изменении курса валюты в настройках себестоимость ранее загруженных позиций пересчитывалась хотя товар уже был закуплен по старому курсу и его стоимость зафиксирована на момент закупки
  • При некорректных данных в одних полях бот давал детализацию по строкам, в других — только общее сообщение без указания конкретных строк
  • Дублирующиеся позиции в выгрузке появлялись многократно вместо корректной агрегации
  • При загрузке двух одинаковых товаров с разной ценой бот отображал их как один но брал цену из более старой записи
  • Временные метки в выгрузках не соответствовали фактическому времени операции
  • Кнопка отмены в ряде сценариев не отменяла уже введённые данные — после нажатия документ всё равно загружался и попадал в статистику
  • Необязательные поля не давали возможности пропустить шаг — кнопка пропуска не реагировала
  • Навигация "Назад" в многошаговых сценариях не работала, сбросить можно было только через полную отмену
  • Файлы с переименованными или англоязычными столбцами принимались без ошибок вместо того чтобы отклоняться
  • Пустые обязательные поля в одних разделах обрабатывались с пояснением, в других — пропускались без сообщения об ошибке
  • При отправке файла не того формата бот зависал без сообщения об ошибке
  • Единственная кнопка в разделе администратора не реагировала на нажатие
Взаимодействие с разработчиком

Простые баги передавала сразу в переписке. Сложные оформляла структурированно: шаги воспроизведения, ожидаемый и фактический результат, скриншот или видео.

Часть поведения в процессе оказывалась не багами — некоторые сценарии были задуманы именно так, часть ограничений объяснялась особенностями Telegram. Разработчик пояснял, я фиксировала и не возвращалась к этому повторно.

Результат

Основные сценарии работают стабильно. В процессе итерационного тестирования выявлен и передан разработчику полный список проблем: валидация входных данных, корректность расчётов, навигация, обработка граничных случаев.

Критичные баги устранены. По части поведения получены пояснения от разработчика — часть ограничений обусловлена особенностями платформы Telegram.
